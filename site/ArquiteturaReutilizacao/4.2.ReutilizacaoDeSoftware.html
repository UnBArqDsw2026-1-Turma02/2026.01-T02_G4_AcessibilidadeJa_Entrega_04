    4.2. Módulo Reutilização de Software

    Conceito: Reutilização de Software via Framework

    Reutilização de software é a prática de construir sistemas a partir de partes já existentes em vez de escrever tudo do zero. Uma das formas mais expressivas de reuso é o framework: um artefato semiacabado que já traz a estrutura e o fluxo de controle prontos, deixando para o desenvolvedor apenas os pontos de variação.

    Os dois conceitos centrais de um framework são:

    • Frozen spots (pontos congelados): a parte fixa, comum a todas as aplicações, escrita uma única vez e reutilizada sem alteração.
    • Hot spots (pontos quentes): os pontos de extensão deixados em aberto, que cada aplicação especializa conforme sua necessidade.

    Disso decorre a Inversão de Controle (Princípio de Hollywood — "não nos chame, nós chamamos você"): é o framework quem chama o código da aplicação nos hot spots, e não a aplicação quem chama o framework. O reuso ainda pode ser caixa-preta (a aplicação consome o framework já empacotado, sem conhecer seu interior) ou caixa-branca (a aplicação estende classes do framework por herança).

    No material de referência da disciplina, esses fundamentos aparecem em dois frameworks Java empacotados como .jar (SomaFramework.jar e GenericMathFramework.jar) e consumidos por um projeto Cliente que os referencia no classpath e estende seus pontos de extensão.

    Aplicação no projeto Acessibilidade Já

    O nosso entregável já é, por natureza, um artefato reutilizável: o widget acessibilidadeja.js é um script autônomo (drop-in) que qualquer site pode incluir com uma linha:

    <script src="https://seudominio.com/widget/acessibilidadeja.js" defer></script>

    Mapeando os fundamentos do framework ao projeto:

    Fundamento (framework) No template Java No Acessibilidade Já
    Framework reutilizável SomaFramework.jar acessibilidadeja.js (script drop-in)
    Cliente que reutiliza ClienteFramework (classpath) site que inclui a tag <script>
    Reuso caixa-preta consumo do .jar consumo do .js sem conhecer seu interior
    Hot spot AbstractStrategy.conta() algoritmo de efeito de acessibilidade
    Frozen spot Template.templateMethod() final esqueleto de aplicação do perfil
    Inversão de controle templateMethod() chama o cliente o widget chama o efeito sobrescrito

    Padrões de reutilização aplicados (com código)

    Seguindo a mesma modelagem dos frameworks de referência, o projeto evidencia os padrões em arquivos isolados e documentais dentro de projetocomgofs/public/widget/:

    Strategy — strategy-pattern.js

    O hot spot é a abstração apply(context); cada estratégia concreta o especializa, e o framework opera sobre a abstração sem conhecer a classe concreta (espelha AbstractStrategy.conta()):

    function AbstractAccessibilityStrategy() {}
AbstractAccessibilityStrategy.prototype.apply = function () {
  throw new Error('apply(context) deve ser implementado pela estratégia concreta');
};

function FontScalingStrategy(percent) { this.percent = percent || 120; }
FontScalingStrategy.prototype = Object.create(AbstractAccessibilityStrategy.prototype);
FontScalingStrategy.prototype.apply = function (context) {
  context.html.style.fontSize = this.percent === 100 ? '' : this.percent + '%';
};

// Inversão de controle: o framework chama o código do cliente.
function runStrategy(strategy, context) { return strategy.apply(context); }

    Template Method — template-method-pattern.js

    O frozen spot é o esqueleto applyProfile() (fixo, reutilizado); os hot spots são os ganchos sobrescritos pelo cliente (espelha Template.templateMethod() final chamando operation2()/operation3()):

    AccessibilityProfileTemplate.prototype.applyProfile = function (context) {
  this.reset(context);              // frozen spot
  this.applyPrimaryEffect(context); // hot spot (cliente)
  this.applyExtraEffect(context);   // hot spot (cliente)
  this.persist(context);            // frozen spot
};

    Decorator — decorator-pattern.js

    Já presente no projeto, encadeia efeitos de acessibilidade como decoradores, permitindo compor responsabilidades de forma reutilizável sem alterar o componente base.

    Validação do componente reutilizável (níveis de teste)

    Um componente só é reutilizável com segurança quando há confiança de que ele opera conforme especificado. O material de níveis de teste reforça essa relação com a reutilização:

    • Testes funcionais (caixa-preta): verificam se os componentes operam conforme o especificado para realizar uma funcionalidade — exatamente a garantia que um cliente espera ao reutilizar o widget sem conhecer seu interior.
    • Testes de sistema: comparam o widget integrado a um site com seus objetivos (acessibilidade, usabilidade, desempenho visual).
    • Testes de aceitação: validam, na perspectiva do desenvolvedor que adota a ferramenta, se ela atende ao que foi prometido antes da disponibilização.

    A suíte automatizada do projeto (npm run test, Jest) exercita esses padrões e dá o respaldo necessário para o reuso caixa-preta do widget.

    Demonstração executável (código rodando)

    Para atender ao "mostrar rodando" do template, a equipe disponibilizou uma demo autônoma em demo-reutilizacao/, que sobe back-end + front-end no mesmo lugar, com um único comando e sem dependências externas (apenas o http e o test runner nativos do Node).

    cd demo-reutilizacao
npm start        # back + front em http://localhost:3000
npm test         # 10 testes (Strategy, Template Method, Decorator e serviço REST)

    A demo tem duas páginas:

    • / — o plugin real num site de exemplo. A página inicial é o "Blog Viagem & Café", um site de terceiro comum, sem nenhuma acessibilidade própria. A única alteração feita nele foi uma linha — <script src="/widget/acessibilidadeja.js" defer></script> — e o widget se auto-injeta (botão flutuante + painel). Esta é a evidência central de reutilização: reuso caixa-preta / drop-in em qualquer site. Validado com Chrome headless (o botão ajw-fab e o painel são injetados corretamente).
    • /padroes.html — os padrões isolados (Strategy, Template Method, Decorator) com log de execução, espelhando os padrões internos do widget.

    Por ser um plugin drop-in, o widget também pode ser injetado em sites reais já no ar. A equipe validou isso em Mercado Livre (https://www.mercadolivre.com.br/) e OLX (https://www.olx.com.br/): basta colar um pequeno snippet no console do navegador (ver demo-reutilizacao/README.md, seção "Testar o plugin em sites reais") para o botão e a janela flutuante de acessibilidade aparecerem sobre o site real.
